**Date:** 2026-05-02

## Context — why 1.0.0

Diagnosis trail (recorded for future sessions):

1. User reported the bridge errored when trying gpt-5.5 — same error a parallel Claude session had hit: `"The 'gpt-5.5' model requires a newer version of Codex"`. But `codex` on PATH was v0.128.0 and supported gpt-5.5 fine in the interactive CLI.
2. Reading `node_modules/@openai/codex-sdk/dist/index.js` revealed `findCodexPath()` at lines 365-431 — the SDK resolves a bundled binary via its own `@openai/codex` dep, never consulting `$PATH`.
3. Bundled binary version was 0.104.0 (matching the SDK's exact pin: `codex-sdk@X` → `codex@X`).
4. Direct SDK probe (`temp/probe-sdk.mjs`) confirmed: bundled v0.104.0 + gpt-5.5 fails in 1.4s; user's global v0.128.0 via `codexPathOverride` + gpt-5.5 succeeds in 5s.

So the advertised default model (`gpt-5.5`, set in 0.3.0) had been broken for users on the bundled-binary path since release. 1.0.0 frames the fix.

## Per-commit detail

### Commit 1 — T-002 (atomicity)
- Wrapped `saveReview` + `markSessionCompleted` in `db.transaction(() => { ... })()` in `src/storage/session-tracker.ts`.
- New `src/storage/session-tracker.integration.test.ts` uses a real `:memory:` SQLite with the reviews table deliberately omitted, reproducing the real storage failure path. Exceeds the original ticket acceptance (which only required a mocked-throw test).
- Fixed log-message assertion drift in `src/tools/review-{code,plan,precommit}.test.ts` (was `'Failed to complete session'`, now matches the new transaction-rollback log message).

### Commit 2 — T-001 (multi-chunk session failure)
- Extended `Result<T>` error variant in `src/utils/errors.ts` to carry optional `session_id`. `err()` now accepts an optional second arg.
- Chunk loops in `src/codex/client.ts` (lines 418-441 and 514-528) surface the partial thread id when a mid-chunk failure happens — chunk 1 succeeds creating thread X, chunk 2 fails, error result carries `session_id: X`.
- `recordFailure(sessionId?)` in `src/storage/session-tracker.ts` now branches: preflight path (existing row) is a single UPDATE; fresh-session path opens a transaction to create-then-fail atomically.
- Tool handlers (`src/tools/review-{code,plan,precommit}.ts`) pass `result.session_id` to `tracker.recordFailure()`.
- Two tests: client-level (`src/codex/client.test.ts` — Result.session_id is set) and integration (`src/server.integration.test.ts` — review_status returns `'failed'` end-to-end). The integration test uses **real chunking** via `RB_CONFIG_PATH` + small `max_chunk_tokens` + a generated 2-file diff. Mocks only at the SDK boundary, exercising the actual product path.

### Commit 3 — server.ts version drift
- `src/server.ts` had `version: '0.1.4'` hardcoded since 0.1.x.
- Now reads version once at module load via `readFileSync(new URL('../package.json', import.meta.url))`. Resolves correctly under vitest, the tsup-bundled dist, and npm-installed consumers without changing tsconfig (no JSON module imports needed; `rootDir: "src"` is preserved).
- Regression test in `src/server.test.ts` asserts the advertised version matches `package.json` and is no longer `'0.1.4'`.

### Commit 4 — SDK bump + regression guard
- `package.json`: `"@openai/codex-sdk": "^0.104.0"` → `"0.128.0"` (exact pin, no caret).
- `npm install` regenerated lockfile; pulled `@openai/codex@0.128.0` and the matching platform binaries.
- New `src/codex/sdk-version.test.ts` asserts our SDK dep equals `EXPECTED_SDK_VERSION` (= `'0.128.0'`) AND the SDK's `@openai/codex` dep equals the same value. Future bumps must update both constants together.
- Live SDK probe verified: gpt-5.5 succeeds via the bundled v0.128.0 binary (5121ms, `finalResponse: ok`).
